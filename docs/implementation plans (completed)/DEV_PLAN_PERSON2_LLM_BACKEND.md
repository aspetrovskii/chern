# План разработки для человека 2 (LLM Backend)

Документ задает пошаговый план и обязательные требования к LLM-части проекта `Concert Playlist` в соответствии с `README.md`.

---

## 1) Границы ответственности

- Реализовать модуль работы с YandexGPT:
  - структурирование пользовательского запроса,
  - тегирование треков,
  - мультиязычность входа (ru приоритетно, также en/tr/zh),
  - валидация и нормализация в строгий JSON.
- Реализовать ленивое тегирование с кэшем в SQLite (`tracks_cache`).
- Предоставить стабильный внутренний контракт для человека 3 (optimizer/backend orchestration).
- Не владеть OAuth/Spotify-интеграцией как прод-контуром (только согласованный интерфейс данных).

---

## 2) Пошаговый план разработки

## Шаг 0. Контракт и структура модуля

- Зафиксировать интерфейсы Python-модуля (пример):
  - `parse_user_intent(text, chat_context) -> IntentV1`
  - `tag_track(track_meta, audio_features) -> TrackTagsV1`
  - `tag_tracks_batch(items) -> list[TrackTagsV1]`
- Ввести версионирование схем:
  - `intent_schema_version`,
  - `llm_version` (промпт + модель + правила постобработки).
- Добавить Pydantic-модели для входа/выхода.

## Шаг 1. Structured intent из запроса пользователя

- Вход:
  - текст запроса,
  - метаданные чата (режим, target_track_count, язык UI при необходимости).
- Выход (`IntentV1`) с минимальными полями:
  - `intent_text`,
  - `genres`,
  - `mood_arc`,
  - `energy_target`,
  - `exclude_artists`,
  - `language_detected`,
  - `confidence`.
- Обязательная постобработка:
  - привести синонимы жанров к каноническим значениям,
  - убрать пустые/конфликтующие поля,
  - сохранить исходный текст отдельно для отладки.

## Шаг 2. Тегирование треков (ленивое)

- Вход:
  - `spotify_track_id`,
  - `raw_metadata` (name/artists/album/popularity/duration и т.п.),
  - `audio_features` (tempo, energy, danceability, valence...).
- Выход (`TrackTagsV1`):
  - жанровые и тематические теги,
  - числовые оценки (например mood/drive/lyric-focus),
  - `confidence`,
  - `llm_version`.
- Правило: если в `tracks_cache` есть актуальная запись с тем же `llm_version`, повторный вызов LLM не делать.

## Шаг 3. Retry/timeout/fallback

- Таймаут вызова LLM (например 8-15 секунд, уточнить с человеком 3).
- Ограниченный retry (например 2 попытки с jitter).
- Fallback:
  - если LLM недоступен, вернуть минимальные теги из `audio_features` + эвристик,
  - проставить `confidence` ниже и флаг деградации.
- Все ошибки и деградации логировать структурированно, без PII и без секретов.

## Шаг 4. Безопасность и политика хранения

- Не логировать токены/секреты/полные OAuth-ответы.
- Не сохранять в БД данные, нарушающие ToS (например полные тексты песен без лицензии).
- Хранить только минимально нужные поля для UI и алгоритма.
- Ключи Yandex хранить в env/secret store.

## Шаг 5. Интеграция с API основного бэкенда

- Совместно с человеком 3 закрепить DTO для:
  - `messages.structured_intent`,
  - `tracks_cache.llm_tags`,
  - статусов пайплайна (`tagging`, `optimizing`...).
- Реализовать сервисные методы, которые основной API вызывает внутри `POST /chats/{id}/messages` и `POST /chats/{id}/generate`.
- Добавить интеграционные тесты на поток:
  - parse intent -> cache miss tagging -> cache hit tagging.

## Шаг 6. Качество и мониторинг

- Создать набор 40-60 тестовых запросов (RU/EN/TR/ZH) и эталонных ожиданий по структуре.
- Добавить простые метрики:
  - `llm_latency_ms`,
  - `llm_error_rate`,
  - `cache_hit_rate`,
  - `fallback_rate`.
- Подготовить файл "prompt changelog" (что менялось и зачем).

## 3) Требования к работе части 2 с API проекта

## Что часть 2 обязана принимать от основного бэкенда

- Текст запроса пользователя и контекст чата.
- Нормализованные данные трека от части 3:
  - `spotify_track_id`,
  - `raw_metadata`,
  - `audio_features`.
- Идентификаторы для трассировки (`chat_id`, `message_id`, `request_id`).

## Что часть 2 обязана возвращать в основной бэкенд

- Строго валидный JSON по схеме:
  - `structured_intent` для сообщений,
  - `llm_tags` для `tracks_cache`.
- Признак версии (`llm_version`) для кэш-инвалидации.
- Признак качества (`confidence`) и, при деградации, флаг fallback.

## Нефункциональные требования

- Время обработки одного user intent: целевой p95 до 3 сек (без учета очередей сети).
- Тегирование одного трека: целевой p95 до 2 сек при cache miss; cache hit - мгновенно из SQLite.
- Идемпотентность: повторный вызов с теми же входными данными и `llm_version` не должен порождать дубликаты.

## 5) Критерии приемки (Definition of Done)

- Для входных RU/EN/TR/ZH запросов модуль стабильно возвращает валидный `IntentV1`.
- Ленивая стратегия кэширования работает: при cache hit нет вызова LLM.
- `llm_tags` и `structured_intent` совместимы с пайплайном человека 3 без ручных правок.
- Ошибки LLM не роняют генерацию целиком: fallback активируется контролируемо.
- Все секреты и чувствительные данные защищены; логи чистые.

## 6) Как писать часть 2 с помощью нейросетей (обязательно)

- Не просить нейросеть "сделай весь сервис целиком"; делать малыми задачами:
  - сначала схема,
  - затем валидатор,
  - затем клиент LLM,
  - затем кэш и retry.
- Каждый ответ нейросети проверять тестами:
  - юнит-тесты на нормализацию/валидацию,
  - интеграционные тесты на cache miss/hit.
- Запрещено принимать код от нейросети без:
  - явных таймаутов,
  - retries,
  - обработки невалидного JSON.
- Любое изменение промптов фиксировать в changelog:
  - дата,
  - причина,
  - ожидаемый эффект,
  - риски.
