# План разработки для человека 1 (Frontend)

Документ задает пошаговый план и обязательные требования к фронтенд-части проекта `Concert Playlist` в соответствии с `README.md`.

---

## 1) Границы ответственности

- Реализовать весь пользовательский интерфейс сайта: авторизация, чаты, генерация, редактор пула, ручной порядок треков (drag-and-drop), плеер.
- Интегрироваться с внутренним API проекта (`/api/v1/...`), не обращаться к Spotify Web API напрямую.
- Корректно поддержать два режима воспроизведения:
  - Premium: Web Playback SDK.
  - Non-Premium: кнопка/редирект "Открыть в Spotify".

---

## 2) Пошаговый план разработки

## Шаг 0. Подготовка

- Выбрать стек: React + TypeScript (рекомендуется).
- Создать структуру модулей:
  - `app/auth`
  - `app/chats`
  - `app/concert`
  - `app/pool-editor`
  - `shared/api`
  - `shared/types`
- Вынести базовый HTTP-клиент для внутреннего API, добавить централизованную обработку ошибок `401/403/429/5xx`.

## Шаг 1. Авторизация и сессия

- Экран входа с кнопкой "Войти через Spotify".
- Поток:
  - переход на `GET /api/v1/auth/spotify/login`,
  - обработка возврата на фронт после callback,
  - запрос `GET /api/v1/me`.
- UI-логика:
  - если пользователь авторизован, переход к списку чатов;
  - если нет, показ экрана входа.

## Шаг 2. Список чатов и создание чата

- Реализовать:
  - список чатов (`GET /api/v1/chats`),
  - создание (`POST /api/v1/chats`),
  - открытие выбранного чата (`GET /api/v1/chats/{id}`).
- Для каждого чата показать:
  - название,
  - режим (`fixed_pool` / `spotify_discovery`),
  - целевую длину концерта.

## Шаг 3. Экран чата и генерация

- В чате реализовать:
  - ленту сообщений,
  - ввод запроса пользователя,
  - отправку `POST /api/v1/chats/{id}/messages`,
  - опрос статуса `GET /api/v1/chats/{id}/messages/{msg_id}` до `done/error`.
- При `done` загрузить `GET /api/v1/chats/{id}/concert` и отрисовать карточки треков.
- Показать состояния: `queued`, `tagging`, `optimizing`, `done`, `error`.

## Шаг 4. Настройки чата

- Реализовать редактирование:
  - режима (`fixed_pool` / `spotify_discovery`),
  - `source_spotify_playlist_id`,
  - `target_track_count` (дефолт 10).
- Эндпоинт: `PATCH /api/v1/chats/{id}`.
- Валидация UI: допустимый диапазон длины концерта (например 5..30, финально согласовать с бэком).

## Шаг 5. Редактор пула треков

- Отдельная панель/модалка:
  - выбор плейлистов аккаунта (`GET /api/v1/spotify/playlists`),
  - добавление в пул:
    - отдельных `track_ids`,
    - по `album_id`,
    - по `playlist_id`,
    - по `artist_id` (`POST /api/v1/chats/{id}/pool`),
  - удаление треков из пула (`DELETE /api/v1/chats/{id}/pool/tracks`),
  - пересборка (`POST /api/v1/chats/{id}/generate`).
- Для массовых операций показать прогресс/лоадер и явный статус результата.

## Шаг 6. Ручной порядок (вариант B)

- Реализовать drag-and-drop в финальном упорядоченном списке треков.
- При сохранении отправлять:
  - `PATCH /api/v1/chats/{id}/concert/order`
  - тело: `{ "ordered_track_ids": ["id1", "id2", ...] }`.
- После успешного ответа перезапросить `GET /api/v1/chats/{id}/concert`, чтобы UI совпал с сервером/Spotify.
- При ошибке синхронизации показать `retry` и не терять локальный порядок пользователя.

## Шаг 7. Плеер и fallback

- Проверять флаг Premium в `/me` (или эквивалент, согласовать с бэком).
- Premium:
  - инициализировать Spotify Web Playback SDK,
  - воспроизводить текущий концерт.
- Non-Premium:
  - показывать кнопку "Открыть в Spotify" (deep link на плейлист/контекст).

## Шаг 8. История версий концертов

- Реализовать список версий (`GET /api/v1/chats/{id}/concerts`).
- Минимум в UI:
  - версия,
  - время создания,
  - источник порядка (`optimizer`/`user`).
- Кнопка "Открыть версию" без изменения текущей.

## Шаг 9. Финальная стабилизация

- Проверить пустые состояния, ошибки сети, долгие ответы, повторные клики.
- Унифицировать дизайн и тексты статусов.
- Подготовить документацию для QA: сценарии smoke test.

## 3) Требования к работе фронта с API проекта

## Обязательные правила

- Работать только через backend API проекта, без прямого вызова Spotify/Yandex.
- Не хранить секреты в фронте.
- Все операции, меняющие состояние, должны быть идемпотентно обработаны в UI:
  - блокировка повторной отправки,
  - безопасный `retry`.
- Для длительных задач использовать polling с backoff (например 1s -> 2s -> 3s, до лимита).

## Обязательные эндпоинты

- Auth: `/auth/spotify/login`, `/auth/logout`, `/me`.
- Chats: `/chats`, `/chats/{id}`, `/chats/{id}/messages`, `/chats/{id}/messages/{msg_id}`, `/chats/{id}/generate`.
- Config/pool: `/chats/{id}` (`PATCH`), `/spotify/playlists`, `/chats/{id}/pool`, `/chats/{id}/pool/tracks`.
- Concerts: `/chats/{id}/concert`, `/chats/{id}/concert/order`, `/chats/{id}/concerts`.

## Форматные требования

- Фронт валидирует ответ API через runtime-схемы (например Zod), не доверять "сырому" JSON.
- Для `PATCH /concert/order` фронт обязан отправлять полный список `ordered_track_ids` без пропусков и дублей.
- При любом `error` от API показывать пользователю человекочитаемое сообщение + технический код для дебага.

---

## 4) Критерии приемки (Definition of Done)

- Пользователь может войти через Spotify, создать чат и отправить запрос.
- При генерации корректно отображаются статусы `queued/tagging/optimizing/done`.
- Виден упорядоченный концерт, работает drag-and-drop и сохранение порядка.
- В Premium-режиме проигрывание в сайте работает; без Premium доступен рабочий deep link.
- Редактор пула поддерживает добавление/удаление и пересборку.
- История версий концертов доступна и читаема.
- Нет утечек секретов/токенов в UI/логи.

---

## 5) Как писать фронт с помощью нейросетей (обязательно)

- Генерировать код небольшими инкрементами (1 экран/фича за итерацию), а не "весь фронт сразу".
- Каждый сгенерированный фрагмент проверять вручную:
  - типы,
  - обработка ошибок,
  - соответствие API-контракту.
- Не принимать код, который:
  - вызывает Spotify напрямую,
  - хранит токены в `localStorage` без решения команды,
  - не обрабатывает `loading/error/empty`.
- Для каждого merge request прикладывать:
  - список измененных API-эндпоинтов,
  - скриншоты основных состояний,
  - чеклист пройденных сценариев.
